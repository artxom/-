# DWS 造数工具 V3.0 规划书：高级造数模板与强关联数据生成

## 1. 业务目标与背景
当前的 V2.0 版本（Milestone 2）已经圆满完成了持久化知识库管理、长效记忆注入以及全自动化的大模型交互数据统计大盘功能。

**V3.0 目标**：在数据生成阶段，单纯依赖大模型随机生成数据往往无法满足真实业务系统严格的约束。因此，V3 的核心目标是**强化数据的合法性与业务关联性**，引入内置的造数模板引擎。

## 2. 核心需求与场景设计

### 2.1 多表主外键与强关联数据生成
由于用户的底层数据来源往往非常杂乱无章，直接生成散点数据会导致系统联调失败。
- **目标**：引入跨表强关联模板。
- **场景演示**：例如在生成 `orders` 表（订单表）时，它的 `user_id` 必须严格来源于 `users` 表（用户表）已经存在的主键；同时，`orders` 表的 `total_amount`（总金额）可能必须和它对应的明细表 `order_items` 中的单价与数量总和保持严格一致。
- **实现思路**：Agent 需具备读取并识别“主外键约束”的能力。在生成插入语句时，优先采用 `INSERT INTO ... SELECT ...` 的形式从已有的关联表中抽取随机记录，确保数据链条的完整性。

### 2.2 字段级精细化内容生成 (Prompt-driven Field Generation)
用户需要更颗粒度的数据定制能力，而不是笼统地对整张表下发指令。
- **目标**：提供字段级别的自定义生成规范。
- **场景演示**：在页面上选择 `users` 表的 `email` 字段，并在旁边直接输入规范提示词：“请确保所有的邮箱后缀都是 `@company.com`，并且前缀由小写英文字母和数字组成”。
- **实现思路**：在前端针对表结构的展示面板中，为每个字段新增一个“配置规范”的输入框。这些细粒度的规范将作为结构化的约束条件，被 Agent 在解析时单独注入给大模型，确保大模型生成的 SQL `Mock` 数据完全符合该字段的定制提示词。

## 3. 架构设计与实现思路 (给下一次会话的指引)

如果你（下一个 AI Agent）看到了这份文档，说明你正在着手开发 V3.0 版本。请参考以下步骤：
1. **分析现状**：目前的造数能力强依赖 Agent 自己推断，你需要打破这个黑盒，在前端提供更加直观的表结构树 (Schema Tree) 或字段级别的表单。
2. **拓展后端接口**：在 `backend/main.py` 中，可能需要新增类似 `GET /api/db/schema/{table}/fields` 的接口，方便前端拉取并让用户对单个字段进行点击和编辑。
3. **改造 Prompt 组装引擎**：在 `backend/agent.py` 中，支持接收结构化的 `field_constraints`（字段约束字典）。当调用 `propose_modification` 时，强制大模型应用这些字典中的规则。
4. **主外键探索工具**：给大模型提供一个新的工具函数 `get_table_relations(table_name)`，使其在造数前能主动探明该表的外键依赖，从而生成具备关联查询逻辑的 SQL 脚本。
