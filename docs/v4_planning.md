# DWS 造数工具 V4.0 规划书：篡·数据修改核心功能设计

## 1. 业务目标与背景
在完成了 V3.0 的 Schema 探索器与结构连通性验证后，我们发现“基础造数”页面其实更适合用于进行“有规则的数据修改与对齐”，而非单纯的随机数据插入。

因此在 V4.0 里程碑中，原有的【基础造数】将被更名为【篡·数据修改】。该页面将精简为两大核心场景，旨在通过直观的 UI 与大模型协作，实现复杂场景下的数据刷新与修正。

## 2. 核心功能设计

### 功能一：双表连线与等价键对齐更新
解决因为脏数据或系统迭代导致的不同表中关联字段不一致的问题。

- **交互逻辑**：
  1. **主附表选择**：用户在界面上选择一张“主表”（通常是核心表，数据质量高）和一张“附表”（需要被 UPDATE 修改数据的表）。
  2. **键值匹配**：系统列出两张表的字段，用户指定用于匹配的键（可以是主键 PK、外键 FK，或是具有等价业务含义的同名字段）。
  3. **数据更新**：系统根据匹配的键，尽可能多地将“主表”中优质的数据/属性同步更新到“附表”的对应字段中。
- **技术实现**：
  - 后端提供双表关联字段探测能力，帮助用户快速选中可能匹配的字段。
  - 基于用户的连线配置，后端生成诸如 `UPDATE ... FROM ... WHERE ...` 的标准关联更新 SQL 并执行。

### 功能二：AI 驱动的单字段规则更新
解决对于存量数据中某一特定列进行批量清洗或按照新规则重构的需求。

- **交互逻辑**：
  1. **指定目标**：用户在 Schema 中选择某张表的特定字段。
  2. **描述规则**：用户通过输入框向大模型（Agent）自然语言描述该字段的新生成/清洗规则（例如：“将所有空邮箱填充为 username@company.com”或“将所有的状态码转换为大写”）。
  3. **规划与确认**：
     - 大模型收到规则后，**不直接执行**，而是返回该字段即将被修改的“明确规则说明”以及“规划出的 UPDATE 语句草案”。
     - 界面展示大模型的规划结果，由用户进行最终确认。
  4. **执行变更**：用户按键确认后，系统下发 SQL 真正执行修改操作。
- **技术实现**：
  - 调用 `agent.py`，但在 Prompt 层面明确要求 Agent 输出固定格式（包含 rule_explanation 和 sql_proposals）。
  - 前端渲染卡片提供“确认并执行”按钮。

## 3. 下一步计划
V4.0 将彻底淘汰当前页面中仅用于 `INSERT` 的逻辑，转变为一个强大的 `UPDATE` 工作流控制台。开发开始前，需优先重构前端交互布局，分为“双表关联更新”和“AI单列更新”两个 Tab 以隔离不同操作域。
